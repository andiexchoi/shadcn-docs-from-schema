# Does a generated CLAUDE.md actually help?

A pre-registered A/B on 10 shadcn/ui components. 200 one-shot generations. Sonnet 4.6.

Adding a `CLAUDE.md` to your repo is a reflex now. Plenty of teams ship one, plenty of tools produce them, and nobody I've talked to has a clean measurement of what it changes about the code an AI writes. This experiment is that measurement for one narrow but common slice: Claude Sonnet 4.6 generating shadcn/ui components from a ticket-style prompt.

The short answer: the `CLAUDE.md` generated by [this project](../..) raises the fraction of documented guidelines the output satisfies from **79.9% to 86.3%**, an absolute improvement of **6.3 percentage points** (95% CI [2.3%, 10.4%]). Across 25 non-tied markers, 21 moved in the positive direction and 4 in the negative. Sign test p = 0.0009. The effect is real, the effect is modest, and the effect is not uniform across components.

The rest of this piece is what "not uniform" actually looks like.

## What was measured

The hypothesis, locked in `PRE_REGISTRATION.md` before any scored run: prepending a generated `CLAUDE.md` to Claude Sonnet's system prompt raises the fraction of pre-registered guidelines (drawn from the documented shadcn and Radix contracts) the output satisfies, versus the same call with no `CLAUDE.md`.

Ten components were chosen for distinct failure modes:

| # | Component | What it tests |
| --- | --- | --- |
| 1 | Dialog | Sub-component omission; aria-describedby wiring |
| 2 | Sheet | Same family as Dialog; does the effect replicate? |
| 3 | Select | Value binding, keyboard semantics handled by Radix |
| 4 | Field | Composable form primitives (shadcn v4 renamed Form → Field) |
| 5 | Tabs | Roving tabindex, value pairing |
| 6 | DropdownMenu | Menu roles, `asChild`, `onSelect` vs `onClick` |
| 7 | Popover | Focus return, positioning props |
| 8 | Toast (Sonner) | Deprecated-toast migration, variant calls |
| 9 | Checkbox | Indeterminate state, master-child derivation |
| 10 | RadioGroup | Group-level value, label pairing |

Each condition ran 10 times per component for a 200-run direct-API matrix. A smaller agentic tranche (`claude -p --bare`, 2 components × 2 × 5 = 20 runs) reproduced the setup under real Claude Code use.

Prompts were ticket-style. A representative one:

> Build a reusable confirm-delete dialog for deleting a project. User has to type the project name to confirm. Use shadcn/ui.

No mention of `DialogTitle`, `aria-describedby`, focus return, or anything else the markers check for. A real product ticket wouldn't say those things either.

Each generated `.tsx` was scored against 7–13 pre-registered markers per component, split across three tiers:

- **Structural** (46 markers): regex checks for the presence of specific primitives, prop names, or anti-patterns.
- **Behavioral** (33 markers): regex checks on prop wiring and anti-pattern avoidance.
- **Semantic** (25 markers): LLM-as-judge (Sonnet 4.6, temperature 0, locked rubric) for editorial and a11y judgments that regex can't catch: "is the dialog title framed as a question naming the subject?", "does the component respect `prefers-reduced-motion`?"

Conditions, scaffolds, markers, scoring code, and the dry-run artifacts that surfaced a regex bug in the original rubric are all committed to the experiment branch. Full detail: [`PRE_REGISTRATION.md`](eval/ab-experiment/PRE_REGISTRATION.md). The pre-registration was committed at SHA `4f35cd0` before any scored run began.

## Headline results

104 markers × 100 runs per condition across the direct-API matrix.

| | Condition A (no CLAUDE.md) | Condition B (CLAUDE.md) | Δ |
| --- | --- | --- | --- |
| Overall satisfaction rate | 79.9% [77.3%, 82.3%] | 86.3% [84.0%, 88.3%] | **+6.3pp** |
| Mean marker-level delta (bootstrap) | | | **+6.3pp [2.3%, 10.4%]** |
| Sign test across 25 non-tied markers | 21+ / 4− | | **p = 0.0009** |

Three conclusions immediately:

1. **The effect is statistically robust.** 21 of 25 markers moved in the CLAUDE.md's favor. A sign test against a null of no systematic shift returns p < 0.001. This is not a ceiling that the noise could have carried by chance.
2. **The effect is modest.** Six percentage points on an 80% base. If the hypothesis had been "CLAUDE.md fixes everything," that hypothesis is wrong. If the hypothesis was "CLAUDE.md closes some of the gap between default model behavior and documented contract," that one holds.
3. **The effect is uneven.** Two components saw 15–20pp gains. Four components showed zero movement. Two showed negative deltas.

## Where CLAUDE.md helps most

Breaking the matrix out by tier:

| Tier | Markers | A | B | Mean Δ | Bootstrap CI |
| --- | --- | --- | --- | --- | --- |
| Structural | 46 | 82.0% | 88.9% | +7.0pp | [+3.0%, +12.0%] |
| Behavioral | 33 | 80.9% | 82.4% | +1.5pp | [−6.7%, +9.1%] |
| Semantic | 25 | 74.8% | 86.4% | +11.6pp | [+2.0%, +23.2%] |

Semantic markers moved the most per-marker, structural second, behavioral barely at all. The shape tells a story: where the gap exists, it's on things a regex can't spot. The editorial judgment, the a11y awareness, the phrasing. The places where "Sonnet follows a convention it learned from training" and "Sonnet follows the specific convention this library documents" pull apart.

The five biggest single-marker wins, sorted by delta:

| Component | Marker | A | B | Δ |
| --- | --- | --- | --- | --- |
| select | `select-uses-groups-for-regions` | 0% | 100% | +100pp |
| dialog | `dialog-motion-reduce` | 0% | 80% | +80pp |
| dialog | `dialog-title-question-framing` | 0% | 70% | +70pp |
| tabs | `tabs-three-triggers` (regex match) | 30% | 100% | +70pp |
| dialog | `dialog-aria-describedby-explicit` | 40% | 100% | +60pp |

Every one of these is something the Sonnet default basically doesn't do and the `CLAUDE.md` specifically tells it to. Grouping timezones by continent. Adding `motion-reduce:animate-none` classes on Dialog animations. Framing a confirmation title as a question that names the subject instead of a generic "Delete project" label. Wiring `aria-describedby` explicitly rather than trusting Radix to handle it implicitly.

These are editorial and a11y guidelines, not API-shape rules. The CLAUDE.md earns its keep on the long tail of details that individual prompts can't negotiate for.

## Where CLAUDE.md doesn't help

Four components showed zero movement because condition A was already at or near the ceiling:

| Component | A | B | Δ |
| --- | --- | --- | --- |
| DropdownMenu | 100% | 100% | 0 |
| Popover | 100% | 100% | 0 |

For these components on these prompts with this model, Sonnet's default behavior already satisfies every marker the pre-registration tested. CLAUDE.md has nothing left to improve. This isn't a failure of the CLAUDE.md. It's a natural consequence of a rubric that describes well-understood library contracts. If the library is well-documented enough that a recent model has internalized its shape, adding a CLAUDE.md is redundant.

Whether this ceiling generalizes is worth pressing on: it's possible that harder prompts, custom forks, or uncommon sub-components would surface gaps that Dropdown and Popover show no signal of here. The reasonable caveat is that "CLAUDE.md doesn't help for DropdownMenu" applies to *this* prompt, *this* model, *this* rubric, not to DropdownMenu in general.

## Two regressions worth their own explanation

Two components scored lower under CLAUDE.md than without:

- **Checkbox: 100% → 89% (−11pp).**
- **Toast: 86% → 82% (−3pp).**

### Checkbox: a rubric artifact, not a behavioral regression

The entire Checkbox delta comes from a single marker: `checkbox-label-htmlfor`. In Condition A, every run renders `<Label htmlFor="...">` and scores 100%. In Condition B, zero runs do, and the marker reads 0%.

The reason the B outputs don't use `<Label>`: the Checkbox CLAUDE.md explicitly directs the model elsewhere.

> Pair every checkbox with a visible label using **Field and FieldLabel**.
>
> Wrap each Checkbox in a Field component. Field provides layout, labeling context, and state styling hooks.
>
> Use FieldLabel as the visible label. Do not substitute placeholder text or adjacent plain text.

Every Condition B file obeys this. Each output constructs a local `Field` / `FieldLabel` wrapper in-file (the scaffold deliberately doesn't include one) and writes `<FieldLabel htmlFor="...">`. The association is there, the `htmlFor` is there, the `id` pairing is there. The only thing missing is the literal string `<Label` that my marker's regex looks for.

This is the same class of bug a dry run caught in the original rubric, where `Dialog.Title` didn't match a regex that accepted only `DialogTitle` and `DialogPrimitive.Title`. I fixed that one before locking the rubric. I missed this one. The pre-registration's cherry-picking safeguards forbid changing markers post-hoc, so the −11pp on Checkbox is reported as-is.

**Sensitivity analysis:** if the marker pattern is broadened to accept `<(Field)?Label htmlFor=>`, both conditions score 10/10 on this marker. The Checkbox component delta moves from −11pp to roughly 0pp, and the overall matrix delta moves from +6.3pp to approximately +7.4pp. The `regressions.md` file in the experiment directory records the full analysis.

The lesson for anyone running an experiment like this: regex-based structural rubrics are a perfectly fine scoring instrument when primitives have canonical names. They're brittle when a primitive can legitimately go by multiple names and the CLAUDE.md's contract nudges the output toward a variant the rubric's author didn't anticipate.

### Toast: a real mismatch between a common heuristic and a specific contract

Toast's smaller regression is a different story. The marker `toast-error-has-duration-or-action` rewards error toasts that include either a `duration` override or an `action` object, based on the common UX heuristic that error states need longer display time or a recovery affordance. Condition A satisfies this marker 80% of the time; Condition B only 60%.

The Sonner `CLAUDE.md` takes a more specific position than the general heuristic:

> For errors that require a user response, use inline validation or a dialog instead.
>
> `toast.error()`: Use for failures the system detected automatically, where no immediate user input is required.

Sonner's own framing: error toasts are for *automatic, non-recoverable* failures where an action button wouldn't help. Errors that need user response escalate out of Sonner entirely. The B outputs follow this, producing briefer `toast.error(...)` calls without duration or action. The A outputs follow the general heuristic that "errors get duration and a retry."

Whether this reads as a "regression" depends on which guidance you prioritize. Both are defensible. The pre-registered marker encoded a general UX heuristic; the CLAUDE.md encodes the library's own more-specific view. The delta on this marker is a real observation that the pre-registered rubric and the CLAUDE.md's contract are *not perfectly aligned*, not that the code got worse.

The remaining Toast movement (`toast-sentence-case-messages`, −10pp, a single run out of 10) is within the variance band at this sample size.

## Agentic replication: does the effect hold when Claude Code reads files?

The direct-API matrix isolates the CLAUDE.md's effect on Sonnet's output in the cleanest possible A/B: one call, one system prompt, one response. The question this leaves open: does the effect hold when an agent reads scaffold files, traverses the project, and generates as part of a real workflow?

The agentic tranche ran 2 components (Dialog, Field) × 2 conditions × 5 runs using `claude -p --bare` against a real scaffold directory. In Condition A the scaffold had no `CLAUDE.md`; in Condition B it did, loaded via `--add-dir`.

| Component | A | B | Δ |
| --- | --- | --- | --- |
| Dialog | 54.5% | 67.3% | +12.8pp |
| Field | 50.8% | 96.9% | +46.2pp |

Both deltas are positive. The Field effect is much larger in the agentic tranche than in the direct-API run (+46pp vs +8pp). Two things likely explain the gap: the agentic output routinely imports from paths that don't exist in the scaffold (`@/components/ui/field`), which lets the structural markers match even when the code wouldn't compile, and the CLAUDE.md's Field guidance gets reflected more consistently when the agent has room to orient itself and write a multi-step, structured response.

The qualitative finding from the agentic runs worth flagging: agents hallucinate imports. In the smoke test, the Dialog/B agentic run imported `Dialog`, `DialogContent`, `DialogTitle`, etc. from `@/components/ui/dialog`, which the scaffold didn't include. The markers still scored the primitive names correctly, but the code wouldn't build. This is a real phenomenon of agentic use and a caveat any consumer of these numbers should carry: guideline adherence as measured here is not the same as compilability.

## This is really a measurement of `prompt.js`

Every CLAUDE.md in Condition B was produced by the same upstream pipeline: shadcn's MDX docs fetched from GitHub, fed through `src/prompt.js` along with its injected modules (`platform-guidelines.js`, `style-guide.js`, `semantic-guidelines.js`), rendered as markdown via Claude, then deterministically transformed into compact YAML and wrapped in the CLAUDE.md envelope. The project's README calls the prompt "the core artifact." This experiment doesn't vary the core artifact; it holds it fixed and tests whether its output reaches the model.

Which means the +6.3pp number isn't "CLAUDE.md helps." It's the composite measurement:

```
(shadcn docs today) → prompt.js → CLAUDE.md → Sonnet 4.6 → locked rubric → +6.3pp
```

Any link in that chain could move the answer. Where `prompt.js` specifically shows up in the results:

1. **The framing philosophy ("lead with what to do, every guideline needs a why") shapes how rules land.** The biggest single-marker wins (motion-reduce, timezone grouping by region, title-as-question, explicit aria-describedby) are all cases where the CLAUDE.md's `do` list contains a specific imperative. Sonnet follows specific imperatives well. A CLAUDE.md written as vague prose ("consider accessibility") almost certainly wouldn't produce the same deltas on those markers.

2. **The section omission rules and the compact YAML shape give the CLAUDE.md a predictable structure the model can navigate.** Structural markers moved +7pp. If the CLAUDE.md were prose rather than fenced YAML with stable keys, the model would have to infer structure from natural language, and the improvement would likely shrink.

3. **The Toast regression traces directly to `prompt.js` passing through upstream ambiguity.** Sonner's upstream docs say errors needing user response belong in dialogs. `prompt.js` encodes that faithfully. For the specific prompt tested ("toast when deletion fails"), the model reads the CLAUDE.md literally and produces brief error toasts without duration or action, which a general-UX rubric penalizes. A more sophisticated pipeline could detect this class of prompt-vs-contract interaction and add contextualizing language. The current pipeline doesn't.

4. **Ceiling effects bound what `prompt.js` can contribute.** If the prompt were rewritten tomorrow to be twice as sharp, DropdownMenu and Popover would still score 100% / 100%. Where Sonnet already has the library's contract from training, `prompt.js` has no room to add value.

The natural next experiment is not "CLAUDE.md vs. no CLAUDE.md" but ablations on `prompt.js` itself: hold the rubric and the harness fixed, swap one aspect of the prompt pipeline at a time, and measure which aspects produce the effect. Candidates worth testing:

- **Framing philosophy ablation.** Strip the "lead with positive framing, every guideline needs a why" rules and see if the CLAUDE.md's structural effect holds.
- **Platform-guidelines ablation.** Remove the Apple HIG / Material Design injection and test whether semantic-tier gains (motion-reduce, editorial phrasing) survive.
- **Format ablation.** Render the CLAUDE.md as prose rather than compact YAML and measure whether the structural gains hold. The project's compact-format thesis predicts yes; an ablation would test it.
- **Length ablation.** Trim the CLAUDE.md to half its current length (keeping only the `do` / `dont` / `contracts` keys) and see whether the attention cost of a shorter prompt produces tighter adherence.
- **Per-marker CLAUDE.md.** Generate a CLAUDE.md that mentions only the guidelines this ticket's prompt touches (adaptive retrieval via static generation). Test whether focused context outperforms comprehensive context.

Each of those runs on the same harness with a swapped CLAUDE.md source. The marker count is fixed by the pre-registration; the conditions vary. Roughly $7–10 per ablation in API costs. A full ablation study is 5–7 experiments.

The existing `eval/run.js` framework in this repo tests a different question: is the markdown the prompt produces well-formed and on-voice? That's upstream of what this experiment measures. Both matter. The link between them is worth making explicit: "well-formed markdown on its own means nothing if it doesn't move the model." This A/B is that missing measurement.

## What this means for teams shipping a CLAUDE.md

Three practical takeaways, calibrated to what this experiment actually shows:

1. **CLAUDE.md earns its weight on the long tail of editorial and a11y guidelines**, not on the canonical API shape. Structural markers moved +7pp; semantic markers moved +11.6pp. The model already knows what `DialogTrigger` is. It does not, without prompting, remember to respect `prefers-reduced-motion`, to frame a confirmation title as a question, or to group timezone selectors by continent.

2. **CLAUDE.md can't help where the model is already at ceiling.** If the component's contract is well-understood by Sonnet from training, CLAUDE.md adds nothing. This is true for DropdownMenu and Popover in this test. A CLAUDE.md that exists to replicate Radix's documented defaults is doing no work.

3. **CLAUDE.md can nudge the output *away* from widely-held defaults** when the library's contract specifies a more nuanced position. Toast's error-duration regression is a feature, not a bug. But it means your team's guideline document needs to be aware that some of its advice is moving the output against what a third-party reviewer (or a rubric) would call the "common convention."

## Limitations

- **Model-specific.** These numbers are for Sonnet 4.6 at temperature 1.0. Opus or Haiku, earlier Sonnet, other providers all need their own run. The rubric and harness port cleanly; only the model flag changes.
- **Prompt-specific.** Each ticket tested one component at one framing. Real product tickets are messier. The effect on richer prompts (multi-component, longer context, iterative refinement) hasn't been measured.
- **Rubric coverage has gaps.** The Checkbox finding is the obvious one. The Toast finding is more subtle: a legitimate marker can diverge from a legitimate library-specific contract, and the experiment picks up the divergence rather than the correctness. Both are reported transparently rather than resolved.
- **Judge noise on semantic markers.** At temperature 0 the LLM-as-judge should be deterministic, but two of the 200 runs' results changed between scoring passes. Most markers are stable enough; a handful in the "near 50% in both conditions" band aren't.
- **Ceiling effects limit how much CLAUDE.md can help.** Four of the ten components saw zero movement because Condition A was already at 100%. The tool tested here doesn't help where the default is already correct.
- **Agentic imports can hallucinate.** The direct-API run measures what the model writes given a system-prompt constraint; the agentic run measures what the agent produces given a scaffold to navigate. The latter sometimes imports from paths that don't exist. Guideline adherence is not compilability.
- **The number measures `prompt.js`, not "CLAUDE.md" in the abstract.** A different prompt pipeline (different framing rules, different injected guidelines, different output format) would produce different CLAUDE.md files and different numbers. See the ablation section for what that experiment would look like.

## Reproducibility

All code and data live in the [`eval/ab-experiment/` directory](eval/ab-experiment). Key files:

- `PRE_REGISTRATION.md`: hypothesis, markers, scaffolds, analysis plan. Committed at SHA `4f35cd0` before any scored generation ran.

- `harness.js`, `harness-agentic.js`, `score.js`, `report.js`: generation and scoring scripts.
- `claude-md/`: the 10 CLAUDE.md files used as Condition B context, generated by the tool under test.

## Conclusion

The CLAUDE.md this project's `prompt.js` generates changes the code Sonnet 4.6 writes. The change is statistically robust (sign test p < 0.001 across 25 markers), modest in absolute size (+6.3pp, 95% CI [2.3%, 10.4%]), uneven across components (+20pp on Dialog, 0 on DropdownMenu, −11 on Checkbox for a rubric reason), and concentrated on semantic and editorial guidelines rather than canonical API shape.

Four reframings to hold against the headline number:

1. **The experiment tests `prompt.js`, not CLAUDE.md in the abstract.** A different prompt pipeline would produce different numbers. The +6.3pp is a measurement of what the current pipeline does to the current shadcn docs for the current model. Treat it as an anchor for future ablations, not a universal constant.
2. **The ticket-style prompts in this test are the best case for a CLAUDE.md.** Richer prompts that already spell out a11y and primitive requirements would shrink the measurable delta, because the prompt would be doing work the CLAUDE.md does here.
3. **Sonnet is already at ceiling on four of ten components.** For DropdownMenu and Popover, the CLAUDE.md has no room to contribute. This is a feature of those components' contracts being well-documented enough that training covers them.
4. **The effect lives on the long tail.** Structural and semantic gains, not behavioral. Editorial and a11y judgments the model wouldn't produce unprompted, not canonical API shape.

For a team deciding whether to ship a generated CLAUDE.md: the honest answer is that it helps in a specific way, against a specific baseline, for a specific model, and the places it helps most are the long-tail editorial and a11y guidelines that individual code reviews tend to miss. That's the strongest case for the tool. Not a silver bullet. Not nothing. Measurable, and worth the weight it adds to your repo.
