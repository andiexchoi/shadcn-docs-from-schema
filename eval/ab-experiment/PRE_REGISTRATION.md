# A/B Experiment: Pre-Registration

Committed before any generations run. Scope, conditions, components, prompts, markers, scoring method, and analysis plan locked here so nothing gets cherry-picked post-hoc.

## Hypothesis

Prepending a CLAUDE.md generated by this tool to a fresh Claude Code session improves the quality of shadcn/ui component code produced from a realistic ticket-style prompt. "Improves" is measured against a set of pre-registered markers per component, aggregated across structural, behavioral, and semantic tiers.

Direction of expected effect: Condition B (with CLAUDE.md) produces higher marker-presence rates than Condition A (without), on average across markers.

Null: no difference beyond noise.

## Model and harness

- **Model:** `claude-sonnet-4-6`.
- **Primary harness — direct API (option a):** one Anthropic SDK call per run. No tool use, no file reads, no Claude Code system prompt bloat. This is the stricter A/B: the only thing varying between conditions is whether CLAUDE.md content is included in the system prompt.
  - **System prompt (both conditions, base):** short framing that states the project is a Next.js + shadcn/ui + Tailwind project; lists the available primitives (`Button`, `Input`, `Label` at `@/components/ui/*`, `cn()` at `@/lib/utils`); lists the relevant Radix/RHF deps that are installed per task; instructs the model to output ONLY the file contents of the target `.tsx`, with no prose, no markdown fences, no explanation.
  - **Condition B only:** CLAUDE.md content from `node src/batch.js --components <component> --combine claude` is appended to the system prompt, labeled clearly as "Project component guidelines."
  - **User message (both conditions):** the ticket-style prompt + a one-line "Write the implementation as `components/<slug>.tsx`." directive (slug pre-specified so output path is unambiguous).
  - **Temperature:** 1.0 (API default). Same in both conditions. Noise addressed by n=10.
  - `ANTHROPIC_API_KEY` sourced from the repo's `.env`.
  - Each run's response text is written verbatim as `<run_dir>/components/<slug>.tsx`, with any stray markdown fences stripped if present (regex-safe, logged if stripping occurs).
- **Replication harness — agentic (option b), separate small tranche:** `claude -p --bare --model sonnet --dangerously-skip-permissions --add-dir <scaffold>` running in a real scaffold dir. This tranche exists to validate that the effect observed in the direct-API run holds when agents orient themselves by reading files. Scope is limited (see §Sample size).
- **If a run produces no usable TSX** (empty response, response contains only prose and no file content, or response fails to parse after fence-stripping): record as a failure; retry up to 2× before flagging the run as unscored. Unscored count reported honestly.

## Conditions

- **Condition A:** system prompt = base framing only. No CLAUDE.md content.
- **Condition B:** system prompt = base framing + CLAUDE.md content for the target component (generated by the tool under test).

User message and model parameters are byte-identical across conditions. The *only* varying input is the CLAUDE.md block.

## Sample size

- **Primary (direct-API) matrix:** 10 components × 2 conditions × 10 runs = **200 generations**.
  - Rigorous claim tier uses all 200.
  - Illustrative case-study tier pulls a 5-component × 3-run subset from the same data (first 5 components in list order × runs 1–3 per condition). No new generations.
- **Replication (agentic) tranche:** 2 components (Dialog, Field) × 2 conditions × 5 runs = **20 generations**. Uses the `claude -p --bare` harness against a real scaffold dir, with CLAUDE.md loaded via `--add-dir`. Reports whether the direct-API effect replicates under agentic conditions. Smaller sample; reported as corroborating, not primary.
- **Dry run (pre-matrix):** 1 component (Dialog) × 2 conditions × 2 runs = 4 generations. Cost calibration + harness sanity check. Outputs saved but **not** included in headline stats.

## Components

10 components chosen for distinct failure modes. Order is the evaluation order; also the draw order for the case-study subset.

| # | Component | Failure mode profile |
| --- | --- | --- |
| 1 | Dialog | Sub-component omission (Title, Description); aria-describedby dangling ref |
| 2 | Sheet | Same family as Dialog — sanity check that effect replicates |
| 3 | Select | Value binding (`onValueChange` vs `onChange`); SelectValue placement; keyboard handled by Radix |
| 4 | Field | shadcn v4's composable form primitive (renamed from Form); FieldLabel/FieldDescription/FieldError; htmlFor pairing; aria-invalid + data-invalid pair |
| 5 | Tabs | Roving tabindex; `value` pairing between TabsTrigger and TabsContent; no hardcoded aria-* |
| 6 | DropdownMenu | `asChild` on trigger; `onSelect` (not `onClick`); menu role semantics |
| 7 | Popover | Focus return to trigger; `align`/`side`/`sideOffset` for positioning; no custom Escape/click-outside |
| 8 | Toast | Project's toast primitive API (shadcn Toast or Sonner, whichever the CLAUDE.md reflects); variants; dismissibility |
| 9 | Checkbox | `checked` + `onCheckedChange`; indeterminate state; label association |
| 10 | RadioGroup | Value at group level, not item level; htmlFor/id pairing; no plain `<input type="radio">` |

## Prompts (ticket-style, locked)

One prompt per component. Each is a realistic product-ticket phrasing — does not name required props, a11y, or sub-components.

```
Dialog:       Build a reusable confirm-delete dialog for deleting a project. User has to type the project name to confirm. Use shadcn/ui.

Sheet:        Build a settings panel in a side sheet with a form for notification preferences. Use shadcn/ui.

Select:       Build a reusable timezone selector. Use shadcn/ui.

Field:        Build a profile edit form with name, email, and bio fields. Use shadcn/ui.

Tabs:         Build a settings page with tabs for Profile, Notifications, and Security. Use shadcn/ui.

DropdownMenu: Build a user account menu with Profile, Settings, Billing (disabled if free plan), and Sign out. Use shadcn/ui.

Popover:      Build a color picker popover for a theme editor. Use shadcn/ui.

Toast:        Build a helper that shows a toast when a project deletion succeeds or fails. Use shadcn/ui.

Checkbox:     Build a permissions panel with Read, Write, and Delete checkboxes, plus a "Select all" master checkbox that shows an indeterminate state when some but not all are selected. Use shadcn/ui.

RadioGroup:   Build a billing cycle selector (Monthly, Yearly, Custom) as a radio group. Use shadcn/ui.
```

Output file name convention: `components/<slug>.tsx` where slug is kebab-case of the ticket subject (e.g., `confirm-delete-dialog.tsx`, `notification-settings-sheet.tsx`, `timezone-selector.tsx`). Slug is specified to the agent in a one-line meta-instruction appended to the prompt: `Write the implementation as components/<slug>.tsx`.

## Markers

Markers are locked here. Structural and behavioral tiers scored by regex/AST; semantic tier by LLM-as-judge (Sonnet 4.6, separate call with fixed rubric, temperature 0). Each marker is one of `+` (CLAUDE.md contract supports it), `−` (anti-pattern), or `=` (neutral correctness check).

### 1. Dialog

Structural: `DialogTitle` or `DialogPrimitive.Title` present (+); `DialogDescription` or `DialogPrimitive.Description` present (+); `variant="destructive"` on destructive action (+); no invented variants like `"critical"`, `"danger"` (−).

Behavioral: state via `onOpenChange` only, no `onClick` handler that calls `setOpen(!open)` (+); `aria-describedby` explicit on Content (+); no `placeholder={projectName}` anti-pattern on the confirmation input (−).

Semantic: title framed as a question or action naming the subject, not a generic label (+); UI surfaces the case-sensitivity constraint (+); `motion-reduce` or `prefers-reduced-motion` respected (+).

### 2. Sheet

Structural: `SheetTitle`/`DialogPrimitive.Title` present (+); `SheetDescription`/`DialogPrimitive.Description` present (+); `side` prop used (any value from `top`/`right`/`bottom`/`left`) (+); close button has `aria-label` (+).

Behavioral: state via `onOpenChange` (+); `aria-describedby` explicit (+); no hardcoded positioning override (e.g. `style={{transform: ...}}`) (−).

Semantic: sheet title names what the panel is for (+); form inside the sheet has a submit path that either closes the sheet or explains staying open (+); `motion-reduce` respected (+).

### 3. Select

Structural: uses `SelectTrigger`, `SelectValue`, `SelectContent`, `SelectItem` (all four present) (+); no native `<select>` element (−); placeholder passed to `SelectValue` not `SelectTrigger` (+); `SelectGroup`/`SelectLabel` when options have semantic grouping — timezones have regions (+).

Behavioral: `onValueChange` used, not `onChange` (+); `value` prop on Root; `Label` + `htmlFor`/`id` pairing on the trigger (+); no hand-rolled `role="combobox"` / `aria-expanded` / `aria-haspopup` (−); no custom keydown handlers for arrow/Enter (−).

Semantic: timezone list is realistic (non-empty, IANA-style names like `America/Los_Angeles`) or explicitly parametrized so the caller supplies them (+); disabled state handled or explicitly noted (+).

### 4. Field

*Note:* shadcn/ui v4 renamed the old `Form` (a react-hook-form wrapper) to `Field` — a composable set of primitives (`Field`, `FieldLabel`, `FieldDescription`, `FieldError`, `FieldGroup`, `FieldSet`, `FieldLegend`) that pairs with any form library. The original pre-registration used "Form" before we discovered the upstream rename. Markers updated to target the Field API. This note is part of the pre-registration and is preserved as-is in the final writeup.

Structural: `FieldLabel` present (+); `FieldError` present (+); `FieldDescription` present (+); `Field` or `FieldGroup` wrapper used (+); no raw `<label htmlFor>` (−); schema resolver used (`zodResolver`/etc.) (+); either `react-hook-form` imported or an explicit uncontrolled `<form onSubmit>` pattern (+).

Behavioral: `FieldLabel htmlFor="X"` is matched by an `id="X"` on a sibling input within 400 chars (+); when error state is rendered, `aria-invalid` on the control is paired with `data-invalid` on Field within 400 chars (+); submit button reflects form-pending state (`isSubmitting`, `isPending`, `isLoading`, or equivalent) (+).

Semantic: submit handler strongly typed from schema inference (+); validation covers email format + bio max + name min-or-nonempty (missing one ok; missing two+ not) (+); `FieldDescription` rendered with real helper content on at least one field (+).

### 5. Tabs

Structural: `Tabs`, `TabsList`, `TabsTrigger`, `TabsContent` all present (+); each trigger has a `value`; each content has matching `value` — pair count equals trigger count (+); `defaultValue` or controlled `value`+`onValueChange` (+); no `role="tab"` / `aria-selected` / `tabindex` hardcoded (−).

Behavioral: no custom keydown handlers for arrow navigation (−); no unmounting logic that discards tab state unless intentional and commented (+).

Semantic: each tab's `TabsContent` has meaningful content or a clear scaffold placeholder per section (Profile / Notifications / Security) (+); icon-only tabs include accessible text (n/a expected for this prompt).

### 6. DropdownMenu

Structural: `DropdownMenu`, `DropdownMenuTrigger`, `DropdownMenuContent`, `DropdownMenuItem` all present (+); `asChild` on `DropdownMenuTrigger` when wrapping a Button (+); `DropdownMenuSeparator` separating logical groups (+); `DropdownMenuLabel` for section title if group is labeled (+).

Behavioral: `onSelect` on items, not `onClick` (+); `disabled` prop on Billing item when free plan, not conditional render (+); no `role="menu"`/`role="menuitem"` hardcoded (−).

Semantic: Sign out item is visually distinguished (destructive variant, color class, or explicit separator above) (+); free-plan check is a prop or context, not hardcoded (+).

### 7. Popover

Structural: `Popover`, `PopoverTrigger`, `PopoverContent` present (+); `asChild` on trigger when wrapping a Button (+); `align`/`side`/`sideOffset` used for positioning, not inline `style` transforms (+).

Behavioral: state via `open`/`onOpenChange` (+); no custom Escape handler (−); no custom outside-click handler (−); focus returns to trigger on close — Radix default, so *not* manually overridden (+).

Semantic: color picker renders either a swatch grid or a real picker (not a text input labeled "color") (+); selected color is a controlled value with `onChange`/`onValueChange`-style API (+).

### 8. Toast (Sonner)

Structural: imports `toast` from `sonner` (+); uses `toast.success()` for success and `toast.error()` for failure — not a single `toast()` with manual styling (+); no home-rolled toast (`useState([...])` with custom render) (−); does NOT import from `@radix-ui/react-toast` (−, since shadcn migrated off it).

Behavioral: error toasts pass `duration` override (longer than default, or `Infinity` with explicit dismiss) (+); error toasts include an action object (`{ label: "Retry", onClick }`) OR a dismiss affordance (+); success toasts are terse, error toasts include enough context to recover (+).

Semantic: two distinct call sites or a single helper that branches on result type (+); messages follow sentence case and name the outcome ("Project deleted," not "Success") (+); a `<Toaster />` mount is mentioned in a code comment or JSDoc even if not included in the same file (so the consumer knows it needs to exist once at app root) (+).

### 9. Checkbox

Structural: `Checkbox` from the Radix/shadcn primitive, not `<input type="checkbox">` (−); `Label` with `htmlFor` + matching `id` on every checkbox (+); master checkbox supports `checked="indeterminate"` (+).

Behavioral: `onCheckedChange` not `onChange` (+); master checkbox derives state from children — `all`, `none`, `indeterminate` — not independent state (+); toggling master selects/deselects all children atomically (+).

Semantic: labels are human-readable ("Read", "Write", "Delete" — not "read_perm") (+); state is lifted to a parent and passed down, not three independent `useState`s unless explicitly required (+).

### 10. RadioGroup

Structural: `RadioGroup` + `RadioGroupItem` primitives, not `<input type="radio">` (−); every item has a unique `value`; every item has `id` matched by `htmlFor` on an adjacent `Label` (+).

Behavioral: `value`/`onValueChange` on `RadioGroup` root, not on individual items (+); `name` inherited via RadioGroup (not manually set on each item) (+).

Semantic: "Custom" option reveals or hints at additional input — either inline revealed input, a second field, or a comment noting the caller wires it (+); default value is a sensible pick (Monthly or Yearly, not Custom) (+).

## Scoring method per tier

- **Structural:** regex pattern over the file text. Pattern + occurrence count rule per marker listed in `eval/ab-experiment/scorers/structural.js`. Deterministic.
- **Behavioral:** more complex regex (multi-line, AND/OR across patterns) or small AST checks via `@babel/parser`. Deterministic. Listed in `scorers/behavioral.js`.
- **Semantic:** single API call per file to Sonnet 4.6, temperature 0, with a fixed rubric prompt that asks for a JSON object `{marker: boolean, ...}`. The rubric is written and locked *before* any run. Listed in `scorers/semantic.js`. One call per run file → ~220 additional API calls (200 primary + 20 replication). Noise bounded by the LLM-as-judge's consistency at temperature 0 (spot-checked manually on a handful of outputs before committing the rubric).

Each marker scored `+1` if satisfied, `0` if not, `NaN` if n/a for that prompt (e.g. icon-only tab check when the prompt has no icon tabs — scored n/a, not 0).

## Analysis plan

Per marker, per condition, compute **proportion satisfied across 10 runs** ($\hat p_A$, $\hat p_B$).
- Report $\hat p_B - \hat p_A$ per marker.
- Report 95% Clopper–Pearson CI on each proportion.
- Report paired comparison across all markers: sign test on the difference vector ($H_0$: median difference = 0), and mean effect across markers with bootstrap 95% CI (1000 resamples over marker-level differences).
- High-variance markers (proportion between 0.3 and 0.7 in *either* condition) explicitly flagged as noisy.

Pre-specified headline number for the blog post: **"Across N markers × 10 components, CLAUDE.md increased average marker satisfaction rate from $\bar p_A$ to $\bar p_B$, an absolute improvement of $\Delta$ (95% CI [lo, hi])."**

No marker is re-weighted post-hoc. No marker is dropped post-hoc unless it's flagged as n/a for structural reasons (e.g. Semantic marker "icon-only tab includes accessible text" when the generated Tabs contain no icon-only triggers — in that case every run in both conditions is n/a, so the marker contributes no data).

## Pre-lock revisions (2026-04-18)

This section records every change to components, prompts, markers, or methodology made *before* the first scored generation ran. Each revision is dated and justified. After this file is committed (see §Outputs and artifacts), nothing below this section is editable.

### 2026-04-18 — Form → Field

**What changed:** swapped component 4 from `Form` to `Field`.

**Why:** shadcn/ui v4 renamed the old `Form` (react-hook-form wrapper) to `Field` — a composable primitive set (`Field`, `FieldLabel`, `FieldDescription`, `FieldError`, `FieldGroup`, `FieldSet`, `FieldLegend`) that pairs with any form library rather than being tied to RHF. The upstream `form.mdx` no longer exists, so the generator produced no CLAUDE.md for it. Markers rewritten to target the Field API as documented in the generated `field.md`. Prompt text unchanged.

### 2026-04-18 — Patched namespace-dot regex in structural markers

**What changed:** every structural marker that checks for a Radix primitive's presence now accepts three shapes: the shadcn wrapper name (`DialogTitle`), the `Namespace.Component` form (`Dialog.Title`, `DialogPrimitive.Title`), and any `<X.Title>` where X is a capitalized identifier. Compound "all four present" markers for Select, Tabs, DropdownMenu, Popover, and RadioGroup were split into atomic one-primitive-each markers.

**Why:** a dry run (1 component × 2 conditions × 2 runs, Dialog only) surfaced that the original regex only matched the shadcn wrapper and the `Primitive.`-prefixed form. Condition A outputs frequently used `import * as Dialog from "@radix-ui/react-dialog"` and wrote `<Dialog.Title>`, which the original regex missed. This produced a false "marker failed" in Condition A for a marker that was actually satisfied in the source, systematically favoring Condition B.

**Evidence:** the dry run's dialog/A condition scored 36.4% satisfaction under the original markers; re-scoring the same files with the patched regex produced 75.0%. The inflated delta (+36.3 pp) under the original markers was substantially a scoring artifact. The delta under patched markers on the same 4 dry-run files is +12.5 pp (structural + behavioral only; semantic not re-scored here).

**Methodological note:** using dry-run data to patch the rubric is legitimate when the change is mechanically justified (a regex bug that fails on a syntactically-valid alternative form) and documented *before* any primary-matrix generation runs. The dry-run files (`eval/ab-experiment/runs/direct/dialog/{A,B}/run-{01,02}.tsx`) and both scored.json variants are retained in-tree as evidence that the patched regex does not uniquely favor either condition. The dry run itself remains excluded from the headline numbers per §Cherry-picking safeguards item 7.

**Marker count changes as a result of the split:**
- Select: 11 → 14 markers (+3, from splitting `select-primitives-all-four` into four atomic markers)
- Tabs: 9 → 11 markers (+2, splitting `tabs-primitives-all-four`)
- DropdownMenu: 8 → 10 markers (+2, splitting `ddm-primitives-all-four`)
- Popover: 8 → 9 markers (+1, splitting `popover-primitives-all-three`)
- RadioGroup: 8 → 9 markers (+1, splitting `radio-primitives-used`)
- Dialog, Sheet, Field, Toast, Checkbox: unchanged

### 2026-04-18 — Toast locked to Sonner

**What changed:** scaffold dep list reduced from `@radix-ui/react-toast + sonner` to `sonner` only. Markers rewritten to be Sonner-specific (`toast.success` / `toast.error` / `duration` / `action` / `<Toaster />`).

**Why:** shadcn/ui deprecated its in-house Toast and now documents Sonner as the replacement. Keeping both packages invited ambiguity. The generated CLAUDE.md for the "sonner" shadcn doc is what's being tested, so the scaffold should reflect that.

## Cherry-picking safeguards

1. This file is committed to git before any generation run. The git SHA is recorded with the results.
2. `results/raw.json` is committed exactly once, at the end of the full matrix run. Partial results are not used to adjust markers.
3. Scoring code is committed before generations run. Running the scorers on the generated data cannot alter the rubric.
4. If a marker turns out to be ambiguous after seeing outputs, it is NOT dropped. It is flagged in the writeup and reported as-is. An addendum can propose new markers for a future run, but cannot alter this one.
5. Run numbering is 1–10 per cell (1–5 for the replication tranche); no sampling, no discarding except for harness-level failures (no usable TSX after 3 attempts), which are reported honestly.
6. Temperature 1.0 (Sonnet API default). Seeds not set — expected source of noise, addressed by n=10 per cell.
7. The dry run (1 component × 2 × 2) is explicitly excluded from the headline numbers. It exists only to calibrate cost and sanity-check the harness.

## Budget

- **Primary (direct-API) matrix:** 200 runs × ~3–5k input + ~3k output per run = ~$0.015/run = **~$3 total**.
- **Replication (agentic) tranche:** 20 runs × ~20k input + ~3k output per run = ~$0.10/run = **~$2 total**.
- **Semantic scoring:** ~220 calls × ~$0.005 each = **~$1**.
- **Dry run:** 4 runs = **~$0.06**.
- **Total expected: ~$6–8.** Worst case (larger outputs, more retries) **~$15**.
- `--max-budget-usd 0.50` per run on the agentic tranche caps runaway there. Primary matrix uses SDK `max_tokens` (8192) as its cap.

## Out of scope

- Comparison against other context files (llms.txt, hand-written README, etc.).
- Comparison against other generators.
- Comparison with `components/ui/<target>.tsx` pre-installed (Run 1 variant). Keeping scaffold consistent (no pre-installed wrapper) means the measured effect is specifically for the case where the agent has to reconstruct the abstraction — which is the realistic scenario the CLAUDE.md is trying to help with.
- Claude Code versions other than the current local binary (`2.1.114`).
- Models other than Sonnet 4.6.
